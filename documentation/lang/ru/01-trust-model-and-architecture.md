# 1. Модель доверия и архитектура

## 1.1 Общая картина

CryptoWallet — это **прикладная прошивка** на STM32H743. Она не заменяет ROM, option bytes и проверяемый загрузчик. Утверждения о безопасности нужно формулировать **по слоям**:

```
STM32 ROM  →  (опционально) проверяемый загрузчик @ stm32_secure_boot  →  CryptoWallet во Flash
```

| Слой | Репозиторий / владелец | Что доказывается |
|------|------------------------|------------------|
| ROM / блокировка отладки | ST, конфигурация платы | Доступ отладки, RDP, векторы |
| Загрузчик | `stm32_secure_boot` (отдельный репозиторий) | Подлинность образа приложения **до** перехода в app |
| Приложение | **этот репозиторий** | Логика кошелька, проверки в рантайме, поверхности UART/HTTP/USB |

Всё, что требует **криптографии ROM или загрузчика**, **вне области** документации CWUP и HTTP в этом репозитории — они выполняются **после** `main()` и старта планировщика.

## 1.2 Обязанности приложения (этот репозиторий)

- **Подписание транзакций** с подтверждением пользователем на устройстве (кнопка), не по сети.
- **Опциональная проверка целостности прошивки** (`fw_integrity`): CRC/длина по связанной области приложения — выявляет случайные повреждения; **не** verified boot.
- **Диагностика:** UART CWUP (при `USE_RNG_DUMP=0`), HTTP (порт 80), WebUSB — у каждой свои допущения о доверии (см. главы 4–6).

## 1.3 Поверхности (атака / оператор)

| Поверхность | Конфиденциальность | Целостность | Примечания |
|-------------|-------------------|-------------|------------|
| HTTP :80 | Нет (нет TLS) | Разбор по возможности | Только доверенная LAN; подтверждение на OLED |
| WebUSB | Зависит от браузера и кабеля | Бинарный протокол с фреймами | Продуктовый путь наряду с HTTP |
| UART3 CWUP | Нет шифрования | Построчные AT-команды | Лаборатория/CI; **нет** аутентификации на `AT+WALLET?` и т.д. |
| UART3 `USE_RNG_DUMP=1` | Н/П | Сырые байты для статистики | **Отключает** CWUP на этом UART |

## 1.3b Аутентификация и авторизация (явно)

Термины IAM использовать точно: в прошивке **нет** транспортной **аутентификации** (доказательства, какой хост или агент отправил запрос) и детальной **авторизации** (роли, области, политик).

| Вопрос | Что делает дерево исходников |
|--------|-------------------------------|
| **Кто клиент HTTP/WebUSB?** | **Не проверяется.** Нет TLS client auth, API-ключей, разбора заголовка `Authorization` в `task_net.c` и на пути WebUSB. Любой хост в доступной LAN/USB может вызывать эндпоинты. |
| **Кто может инициировать подпись?** | **Не привязано к личности.** `POST /tx` ставит запрос в очередь при успешном разборе/валидации; **устройство** затем требует **физическое нажатие** перед подписью в `task_sign` — это **одобрение человеком**, не криптографическое доказательство личности HTTP-клиента. |
| **Кто может читать `/tx/signed`?** | **Любой**, кто достигает устройства по HTTP — ответы не привязаны к сессии или пользователю. |
| **UART CWUP** | **Нет** аутентификации команд (описано отдельно); лабораторная поверхность. |
| **Доверие к прошивке** | **Другой слой:** проверяемый загрузчик / подпись образа в `stm32_secure_boot` доказывает **целостность кода**, не **личность** пользователя или пира для RPC кошелька. |

**Правило оператора:** считать сетевой путь **неаутентифицированным**; опираться на **размещение в сети** (доверенная LAN), **физический доступ** для подтверждения на устройстве и **просмотр на OLED** — не на секретность канала или идентичность пира.

## 1.4 Структура FreeRTOS

После `HW_Init()` и `osKernelStart()` типичные задачи: дисплей, сеть (LwIP + HTTP), подпись, IO (LED/кнопка), ввод пользователя, опционально USB, опционально CWUP, опционально вывод RNG.

**IPC** (очереди, группы событий, мьютексы) связывает сеть/WebUSB с подписью и дисплеем. Точные дескрипторы — в `wallet_shared.h` и `main.c`.

## 1.5 Флаги сборки, влияющие на безопасность

| Флаг | Эффект |
|------|--------|
| `USE_CRYPTO_SIGN` | Полный путь trezor-crypto vs минимальный только хэш |
| `USE_TEST_SEED` | **Известная мнемоника** — только разработка, не для реальных средств |
| `USE_RNG_DUMP` | Бинарный TRNG на UART; **нет** CWUP на этом UART |
| `USE_WEBUSB` | USB-устройство + дескрипторы WebUSB |
| `USE_LWIP` | Ethernet + HTTP-сервер |

## 1.6 Связь с `stm32_secure_boot`

- Подпись и проверка загрузчика документированы там (например `sign_image.py`, формат образа).
- Тесты на хосте в этом репо (`secure_boot_image.py`, `tests/mvp/…`) проверяют логику подписи/подделки образа **как данные**, не выполнение из ROM.
- Команда CWUP `AT+BOOTCHAIN?` **информационная** (макросы, `VTOR`); не доказывает аппаратный root of trust.
