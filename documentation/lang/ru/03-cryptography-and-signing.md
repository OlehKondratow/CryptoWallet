# 3. Криптография и подпись

## 3.1 Цели

- Выводить ключи из мнемоники BIP-39 (если подпись включена), использовать BIP-32 для путей счетов.
- Подписывать транзакции в стиле Bitcoin с **ECDSA secp256k1** (стек trezor-crypto).
- Использовать аппаратный TRNG STM32 там, где стек ожидает случайные байты; при реализованных проверках — отказ при ошибке.

## 3.2 Режимы сборки

| `USE_CRYPTO_SIGN` | Поведение |
|---------------------|-----------|
| `1` | trezor-crypto: RNG, BIP-39/32, ECDSA, связанные хэши |
| `0` | Урезанный путь (например минимальный SHA-256) — **не** конфигурация аппаратного кошелька |

`USE_TEST_SEED=1` задаёт **известную** тестовую мнемонику — **только** лаборатория/CI; ключи предсказуемы.

## 3.2b Вывод частного ключа (как получаются ключи)

- **Вход сида:** `get_wallet_seed()` заполняет **64-байтовый сид BIP-39** (или возвращает ошибку). В задаче подписи **нет** отдельного пути «только RNG без сида» — корневой секрет это сид.
- **Вывод:** `crypto_derive_btc_m44_0_0_0_0()` в `Core/Src/crypto_wallet.c` вызывает trezor-crypto: `hdnode_from_seed()`, затем путь **BIP-32 `m/44'/0'/0'/0/0`** (Bitcoin по умолчанию в проекте).
- **Детерминированность:** при **фиксированном** сиде частный ключ secp256k1 на этом пути **полностью определён** — на вывод **не** расходуется TRNG. Смена пути или сида меняет ключ.
- **Где TRNG всё ещё важен:** у **ECDSA-подписи** нужна случайность (и внутренности библиотеки могут дергать RNG); **создание новой мнемоники** из энтропии — `crypto_entropy_to_mnemonic_12()` с **128 битами энтропии** (вызывающий должен подать энтропию, например с TRNG в полноценном кошельке). Путь **`task_sign`** только **загружает** сид и выводит ключ — **не** реализует сквозной сценарий «создать новый кошелёк».

## 3.3 Конвейер подписи (концепция)

1. Вход (HTTP `POST /tx` или WebUSB) разбирает получателя/сумму/валюту.
2. `tx_request_validate()` проверяет формат и политику.
3. Запрос ставится в очередь для `task_sign`; пользователь подтверждает кнопкой.
4. `crypto_wallet` выводит ключи и подписывает; ответ через `GET /tx/signed` или завершение WebUSB.

**Оговорка по HTTP:** обработчик может вернуть **200** с HTML даже при ошибке валидации — клиент должен опрашивать статус подписи или доверять OLED.

## 3.4 Гигиена памяти

- **`memzero()`** — затирание чувствительных буферов volatile-записями; компилятор не должен удалять (см. реализацию).
- **Стеки** — фиксированные размеры в конфигурации FreeRTOS; на горячем пути подписи без неограниченного alloc.

## 3.4b Хранение сида / частных ключей (честный масштаб)

- **Точка расширения:** `get_wallet_seed()` (см. `task_sign.c`). Слабая реализация по умолчанию возвращает **ошибку** — встроенного «сейфа» с долговременным хранением ключей в репозитории **нет**.
- **`USE_TEST_SEED=1`:** `wallet_seed.c` подставляет **известную** тестовую мнемонику — только лаборатория/CI.
- **Продакшен** (по комментариям в коде): заменить на secure element, шифрование flash или иную явную модель угроз — вне текущей минимальной интеграции.
- **Разделение:** ключи подписи **образа** загрузчика (`stm32_secure_boot`) и ключи **кошелька** (BIP-32/39) — **разные** секреты и домены доверия.

## 3.5 TRNG и хранение сида с привязкой к плате

Связаны **аппаратная случайность**, **лабораторная проверка** и **будущий** дизайн **зашифрованного сида** с привязкой к плате. Привязка хранилища **не реализована**; TRNG **используется** стеком крипто там, где включено.

### 3.5.1 TRNG в прошивке (сейчас)

- **Аппаратный RNG** (`HAL_RNG` / TRNG STM32) питает **trezor-crypto** и связанный код при включённой подписи; пулы и вызовы API — по ожиданиям библиотеки.
- **Роль:** энтропия для рантайм-крипто (nonce, пути генерации ключей внутри библиотеки и т.д.) — **не** то же самое, что «режим дампа по UART» ниже.

### 3.5.2 Режим дампа по UART (только лаборатория / CI)

- **`USE_RNG_DUMP=1`:** отдельная задача выдаёт **сырые** байты TRNG на UART для **внешних статистических тестов** (например capture + dieharder). В этом режиме **CWUP** на этом UART **не** запускается.
- **Не** считать UART-поток продакшен-путём для **записи** ключей в хранилище: в продакшене вызывать драйвер RNG **внутри прошивки**. Детали и CI: [`06-integrity-rng-verification.md`](06-integrity-rng-verification.md).

### 3.5.3 TRNG и UID в будущем зашифрованном хранилище

| Вход | Типичная роль |
|------|----------------|
| **TRNG (аппаратный)** | **Энтропия** для одноразовых значений: соль установки рядом с шифртекстом, nonce AEAD, любой случайный материал, который нельзя предсказать или скопировать с другой платы. |
| **UID (96 бит, заводской)** | **Привязка:** стабильное значение на кристалл смешивается в **KDF**, чтобы шифртекст + PIN с **этой** flash **не** расшифровывались на **другом** МК. **Не** секрет — см. ограничения ниже. |

Корректный дизайн использует **обе** вещи: случайность с **TRNG**; **UID** (и PIN) против **переносимых** клонов flash. **Не** использовать UID как единственную «энтропию» для обёртки сида.

### 3.5.4 Сид на диске с привязкой к устройству (идея — не реализовано)

**Цель:** хранить **зашифрованный** сид во внутренней или внешней flash, но **привязать** расшифровку к **этой** плате, чтобы тот же шифртекст **нельзя** было открыть на другом устройстве — даже зная PIN и имея образ flash.

**Набросок:**

1. **Уникальный ввод:** чтение **UID** (регистры STM32 `UID`), плюс **случайная соль** при первой настройке (**из TRNG**, §3.5.3), хранимая рядом с шифртекстом.
2. **Вывод ключа:** `K = KDF(PIN || hardware_salt || UID || stored_salt)` (алгоритм и число итераций — отдельный выбор).
3. **На диске:** только **AEAD**-шифртекст + метаданные — **никогда** сырой сид.
4. **Разблокировка:** пользователь вводит PIN; прошивка пересчитывает `K` с **локальным** UID; на другом чипе расшифровка не проходит.

**Чем помогает:** клонированный flash на другой плате; случайное восстановление дампа без явного экспорта.

**Чего не решает:** UID не скрытый root-of-trust; кража **этого** устройства + PIN всё равно даёт расшифровку; перенос кошелька на новое устройство — только через **явный** бэкап/экспорт.

**Совместимость:** с secure element, шифрованием flash или схемой Trezor (PIN + зашифрованный DEK) — привязка к плате как **дополнительный** вход KDF.
