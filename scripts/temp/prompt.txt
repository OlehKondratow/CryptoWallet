Документация команд и фаз готовности: docs_src/UART_PROTOCOL_MVP_en.md (строка CW+READY, AT+PING, AT+FWINFO?, AT+WALLET? и т.д.).

Кратко: что «нужно» в смысле условий
Цель	Условие
WebUSB в прошивке	USE_WEBUSB=1, рабочий USB FS (PA11/PA12), линковка USB-стека и usb_webusb / usbd_desc_cw (уже в Makefile для minimal-lwip).
CWUP по UART	USE_RNG_DUMP=0, физический доступ к USART3 (часто ST-Link VCP), после загрузки дождаться CW+READY.
Оба сразу	Типичная конфигурация CI: USE_RNG_DUMP=0 USE_WEBUSB=1 — они не конфликтуют (разные интерфейсы: USB и UART).
Если задача — не «включить», а расширить (новые команды CWUP, другой протокол по WebUSB, веб-страница под Chrome), уточните, что именно нужно добавить поверх текущего MVP.

Промпт: CWUP-0.1 — фазы A, B, C, D (UART)
Контекст: репозиторий CryptoWallet, протокол CWUP-0.1 по USART3 (лаборатория, CI). Документы: docs_src/UART_PROTOCOL_MVP_ru.md, UART_PROTOCOL_MVP_en.md. Прошивка: Core/Src/cwup_uart.c, USE_RNG_DUMP=0 для текстового CWUP.

Задача: при проектировании кода, тестов и CI явно различать фазы линии и не смешивать режимы.

Фаза A — старт (до готовности CWUP)
До строки CW+READY устройство не шлёт кадры TRNG (бинарные кадры §7).
Допустимы только TEXT: логи/преамбула (если включены), кодировка строк с \r\n.
Хост не считает поток «готовым CWUP» до появления преамбулы приложения после загрузчика.
Фаза B — объявление готовности
Ровно одна строка TEXT с завершением \r\n:
CW+READY,proto=CWUP/0.1,build=<git-describe|unknown>,rng=0|1
rng=1 в строке — если в сборке предусмотрена выдача TRNG по протоколу (не путать с «сырым» USE_RNG_DUMP).
Фаза C — команды (TEXT, режим T)
Хост может слать команды AT+... (§5): AT+PING, AT+CWINFO?, AT+READY?, AT+FWINFO?, AT+BOOTCHAIN?, AT+WALLET?, AT+SELFTEST?, AT+ECHO=, и т.д. по спецификации.
До AT+RNG=START (если/когда реализовано) бинарные кадры TRNG §7 не идут; линия остаётся в TEXT.
CI-типичный сценарий: дождаться CW+READY → опционально маркеры/совместимость → дальнейшие AT.
Фаза D — сессия бинарного TRNG (режим BINARY_RNG, переход T→B)
Начинается только после успешной команды AT+RNG=START,... и подтверждения вида CW+RNG OPEN (когда будет реализовано в прошивке).
TX от устройства — только кадры §7 (magic 0xC7 0x57, длина, полезная нагрузка TRNG, CRC32); в режиме B на этот UART запрещены printf-логи без согласованной блокировки TX.
Завершение: AT+RNG=STOP, таймаут, ошибка RNG, сброс — возврат к TEXT (C/A по смыслу).
Связь с реализацией сегодня:

В коде MVP реализованы A→B→C (TEXT + CW+READY + AT без AT+RNG=START в прошивке).
Фаза D и AT+RNG=START в §11 помечены как не реализованы; альтернатива для сырого потока — отдельная сборка USE_RNG_DUMP=1 (CWUP на UART отключён).
Ограничения безопасности: подпись TX и продуктовые операции — HTTP/WebUSB, не CWUP; UART MVP — диагностика и TRNG по политике.

Текущее состояние (кратко)
Область	Уже есть
UART3	MX_USART3_Init() в hw_init.c (huart3, 115200).
TX	Общий мьютекс UART_Tx_Lock / Unlock: UART_Log (дисплей → лог), cwup_uart.c, при USE_RNG_DUMP — rng_dump.c (CWUP не собирается).
CWUP RX	IRQ HAL_UART_Receive_IT по байту, сбор строки до \r\n, очередь в задачу — не DMA/Idle, не кольцевой буфер в классическом смысле.
AT+ (MVP)	PING, CWINFO?, READY?, FWINFO?, BOOTCHAIN?, WALLET?, SELFTEST?, ECHO= — в cwup_uart.c.
Фазы A/B/C	Состояние линии + CW+READY; фаза D (§7 TRNG) — не реализована.
AT+MARKS / AT+RNG=*	В коде парсера нет (только упоминания в комментариях/доках).
WebUSB	usb_webusb.c + дескрипторы, ping/sign; хост: test_usb_sign.py, udev.
Хост CWUP	test_cwup_mvp.py уже шире минимального списка: после базовых команд идут WALLET, SELFTEST, ECHO, опционально FWINFO+CRC, затем BOOTCHAIN; stress — есть.
Тесты подписи / secure_boot	Отдельно: secure_boot_image.py, test_secure_boot_sign_tamper.py, ecdsa в test requirements — по вашему описанию.
Шаги реализации (по приоритету)
Блок 1 — Закрыть «документируемый MVP» (мало кода)
Фаза D (документация) — В SECURITY_AND_TESTING_RU.md и UART_PROTOCOL_MVP_*: таблица «команда → статус (реализовано / нет / ограничение)», явно: USE_RNG_DUMP = отдельная сборка, CWUP на UART отключён; §7 TRNG и AT+RNG — не реализованы; WebUSB — что именно (pyusb vs браузер).
Согласование с stm32_secure_boot — Зафиксировать в доке: какие поля даёт AT+BOOTCHAIN? сейчас (CW_VERIFIED_BOOT, SCB->VTOR), что не импортируется из загрузчика (нет общего X.509 — как у вас в SECURITY); при необходимости следующий шаг — weak-символы или общий заголовок в обоих репозиториях.
.bin vs Flash — Одна заметка в fw_integrity_check.py / линкер / SECURITY: при каком сценарии прошивки CRC на устройстве должен совпадать с файлом на хосте (уже опираетесь на fw_integrity_snprint() — явно описать границы региона).
Блок 2 — CWUP по UART (прошивка), оставшиеся дыры MVP
AT+MARKS — Реализовать по uart_boot_markers.txt / CI, если маркеры всё ещё нужны без смешения с RNG.
Фаза D (протокол) — AT+RNG=START/STOP, кадры §7, переход s_line_phase в CWUP_PHASE_D_BINARY_RNG, запрет TEXT с того же TX в D; включить CWUP_RNG_FRAMED_AVAILABLE / rng=1 в баннере при готовности.
Приём (опционально, под нагрузку) — Если IRQ+очередь упираются: Idle Line + DMA или кольцевой буфер + парсер строк; сохранить единый TX-мьютекс.
Расширенный список из промпта (сеть, здоровье подсистем, кошелёк, операции) — не смешивать с MVP: отдельная политика безопасности; для поля — только lab-команды с флагами DEBUG/сборки.
Блок 3 — WebUSB
Паритет с CWUP по смыслу — Один развести матрицу: что есть по USB (ping/sign), что только по HTTP/UART — в SECURITY / README.
Браузер — HTTPS landing URL = BOS WebUSB в usbd_desc_cw; страница с navigator.usb и теми же VID/PID.
Тесты — Поддерживать test_usb_sign.py; при необходимости общий «smoke» сценарий с test_rng_signing_comprehensive.py --cwup (если флаг уже есть — выровнять документацию и фактические аргументы).
Блок 4 — Хост и CI
test_cwup_mvp.py — Уже покрывает PING → CWINFO → READY → (FWINFO+CRC) → BOOTCHAIN и stress; при необходимости узкий режим --minimal только под ваш короткий сценарий (без WALLET/SELFTEST/ECHO) — косметика.
CI — cwup-mvp-ci.yml / условные шаги в simple-ci.yml: HIL только при CWUP_SKIP_NO_DEVICE / секретах с портом; без платы — pytest tests/mvp + без UART.
Блок 5 — Долгий хвост (продукт)
Диагностика сети / кошелька по UART — только после threat model и явных guard’ов.
Полноценная проверка «два бинарника в Flash» — стенд stm32_secure_boot + железо (отдельно от хост-тестов формата подписи).
Итог одной строкой
MVP CWUP по UART и базовый WebUSB в репозитории уже в значительной стере реализованы; логичный следующий шаг — документация статуса (блок 1) и явное решение по AT+MARKS / фазе D TRNG (блок 2); WebUSB «как в браузере» и расширения протокола — следующие слои после зафиксированного MVP.
