# HTTP API CryptoWallet

<brief>Минимальный HTTP-сервер на **порту 80** (LwIP, `USE_LWIP=1`). Реализация: `Src/task_net.c`. Аутентификации и TLS **нет** — только доверенная локальная сеть. Подтверждение операции — на устройстве (кнопка), не по HTTP.</brief>

**English:** [HTTP_API_en.md](HTTP_API_en.md)

---

## 1. Общие сведения

| Параметр | Значение |
|----------|----------|
| Базовый URL | `http://<IP-адрес-устройства>/` |
| Порт | **80** (константа `HTTP_PORT` в `task_net.c`) |
| Транспорт | TCP (HTTP/1.1-подобные ответы; полноценный парсер заголовков не используется) |
| Ограничение запроса | Весь запрос должен поместиться в один приём буфера (**&lt; 1024** байт, `REQ_BUF_SIZE`) |
| Таймаут чтения | **2000 ms** (`netconn_set_recvtimeout`) |
| TLS / авторизация | **Отсутствуют** |

IP задаётся DHCP или статикой (см. `main.h`, `app_ethernet_cw`).

---

## 2. Сводная таблица маршрутов

| Метод | Путь | Код | Тип ответа | Назначение |
|--------|------|-----|------------|------------|
| `GET` | `/ping` | 200 | `text/plain` | Проверка доступности (`pong`) |
| `GET` | `/` | 200 | `text/html` | Веб-форма отправки транзакции |
| `GET` | `/tx/signed` | 200 | `application/json` | Статус/последняя подпись (опрос) |
| `POST` | `/tx` | 200 | `text/html` | Приём заявки на подпись (тело см. §4) |
| `GET` | *прочие* | **404** | короткий ответ | Неизвестный путь |
| `POST` | *прочие* | 200 | `text/plain` | Тело **`OK`** (поведение «заглушки»; лучше не полагаться) |

---

## 3. `GET /ping`

**Ответ 200 OK**

```
Content-Type: text/plain
Connection: close
Content-Length: 4

pong
```

---

## 4. `POST /tx`

Постановка транзакции в очередь на подпись (`g_tx_queue` → `task_sign`). После успешной валидации данные дублируются в очередь дисплея.

### 4.1 Заголовки

- Для **JSON**: в запросе должна встретиться подстрока `application/json` (проверка через `strstr` по всему буферу запроса) — обычно заголовок  
  `Content-Type: application/json`.
- Иначе тело обрабатывается как **`application/x-www-form-urlencoded`** (поля `recipient`, `amount`, `currency`).

### 4.2 Поля тела (`wallet_tx_t`)

| Поле | Обязательность | Описание |
|------|----------------|----------|
| `recipient` | Должен пройти проверку после разбора | Строка получателя (до **41** символа + `\0`, см. `TX_RECIPIENT_LEN`) |
| `amount` | Должен пройти проверку | Десятичная сумма, строка (до **23** + `\0`, `TX_AMOUNT_LEN`) |
| `currency` | Опционально | Тикер; если пусто, в парсере подставляется **`BTC`** |

Имена полей в JSON — строго в кавычках: `"recipient"`, `"amount"`, `"currency"`. Парсер JSON **наивный** (`strstr` + поиск пар кавычек), без поддержки экранирования.

**Пример JSON:**

```json
{
  "recipient": "bc1qxy2kgdygjrsqtzq2n0yrf2493p83kkfjhx0wlh",
  "amount": "0.001",
  "currency": "BTC"
}
```

**Пример form (тело):**

```
recipient=bc1q...&amount=0.001&currency=BTC
```

### 4.3 Валидация

Полная семантика: `tx_request_validate()` — см. [tx_request_validate.md](tx_request_validate.md) и `tx_request_validate.h` (коды `TX_VALID_*`).

### 4.4 Ответ HTTP

Во всех случаях после разбора `POST /tx` возвращается **200 OK** с HTML-страницей вида «Transaction submitted…» и редиректом на `/` через ~2 с — **независимо** от того, прошла ли валидация и попала ли запись в очередь. Ошибки при некорректном теле или отклонении валидации **не выделяются отдельным кодом HTTP**; клиент должен опираться на лог устройства или опрос `GET /tx/signed`.

---

## 5. `GET /tx/signed`

**Ответ 200 OK**, `Content-Type: application/json`.

Если подпись готова (`g_last_sig_ready`):

```json
{"status":"signed","sig":"<128 hex-символов ECDSA, 64 байта в hex>"}
```

Если ещё не готово:

```json
{"status":"pending"}
```

Подпись — сырой 64-байтовый блок в hex (формат задаётся в `task_sign` / отображении).

---

## 6. `GET /`

Возвращает одну HTML-страницу с формой (`method=post`, `action=/tx`) и клиентским скриптом, запрашивающим `/tx/signed`.

---

## 7. Ограничения и замечания для интеграции

1. **Один пакет приёма** — очень длинные `POST` могут обрезаться.
2. **Нет различия успех/ошибка по коду ответа** для `POST /tx` — используйте опрос `/tx/signed` или UI на устройстве.
3. **POST не на `/tx`** возвращает `200` и тело `OK` — не используйте как API контракт.
4. Безопасность: только изолированная сеть; для продукта обычно нужны TLS, аутентификация и политика доступа — здесь не реализованы.
