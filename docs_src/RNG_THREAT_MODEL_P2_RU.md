# Модель угроз и практики RNG (P2)

Документ закрывает пункты **P2** из [RNG_SECURITY_IMPROVEMENTS_RU.md](RNG_SECURITY_IMPROVEMENTS_RU.md): документация доверия, side-channel, опциональные расширения.

---

## 1. Доверие к аппаратному RNG (STM32H7)

**Предположение:** используется встроенный периферийный блок **RNG** микроконтроллера серии STM32H7, как описано в **Reference Manual** (RM0433 для STM32H743 и родственных; уточните ревизию под вашу часть).

**Что обещает вендор (уровень документации ST):**

- Генератор на основе **аналоговых источников энтропии** с постобработкой; соответствие **NIST SP 800-90B** (формулировки в DS/AN актуальны на дату выпуска кремния).
- Регистры статуса (ошибки тактирования, готовность данных) должны учитываться в коде (см. P0 в roadmap улучшений).

**Граница доверия:**

- Мы **не** верифицируем кремний независимо; доверие переносится с **ST + выбранной ревизии кристалла**.
- Угроза **бэкдора в RNG на уровне чипа** остаётся вне модели (как для большинства встраиваемых SoC).

## 2. Сценарии отказа и поведение (целевое)

| Сценарий | Риск | Целевое поведение (после P0/P1) |
|----------|------|----------------------------------|
| RNG не инициализирован | Предсказуемый/слабый выход | Не выполнять крипто-операции |
| Ошибка HAL / health | Повреждённый поток | Fail-closed, сброс или отказ |
| Короткое замыкание UART в режиме дампа | Утечка сырья наружу | `USE_RNG_DUMP` только в lab-сборках |

Текущая прошивка может отличаться от таблицы — см. актуальный код `crypto_wallet.c`.

## 3. Side-channel и секреты

**Не делать:**

- Не логировать байты из `random_buffer` / сырой TRNG в UART/printf (кроме явного диагностического `USE_RNG_DUMP` на выделенной сборке).
- Не сохранять дампы RNG на постоянный носитель на устройстве без политики.

**Рекомендации:**

- Критические пути (подпись, деривация ключей) — использовать **trezor-crypto** как есть; не оборачивать секреты в отладочные макросы в release.
- Временные задержки для противодействия анализу мощности — по политике продукта (см. Trezor `random_delays` как идею; в CryptoWallet отдельная реализация не обязательна).

---

## 4. Опциональные расширения (roadmap)

**Secure Element (ATECC608A и т.д.):**

- Политика: либо **дополнительное смешивание** энтропии с TRNG, либо хранение ключей в SE — отдельный проект и модель угроз.

**АЦП / «шум с линии»:**

- Только как **дополнительный** источник с оценкой min-entropy и смешиванием через хэш; не замена TRNG.
