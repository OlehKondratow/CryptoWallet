# CWUP-0.1 — CryptoWallet UART Protocol (MVP)

<brief>Спецификация минимального протокола по UART3 для **лабораторной диагностики**, **CI** и **захвата TRNG** с разделением команд и бинарных данных. Не заменяет основной пользовательский канал (HTTP/WebUSB) и **не** описывает обновление прошивки через UART — это зона загрузчика (`stm32_secure_boot`) и отдельной политики.</brief>

**Сборка по умолчанию (`make` / `minimal-lwip`):** `USE_WEBUSB=1`, `USE_RNG_DUMP=0` — одновременно **WebUSB** (PA11/PA12) и **MVP CWUP** на USART3. Режим `USE_RNG_DUMP=1` отключает CWUP на UART (бинарный TRNG); WebUSB при необходимости отключают: `USE_WEBUSB=0`.

---

## 1. Область применения и границы

| Входит в MVP | Не входит в MVP (явно) |
|----------------|-------------------------|
| Текстовый канал команд после старта приложения | Протокол ROM-bootloader / ST-Link |
| Кадрированная выдача TRNG для dieharder | DFU/OTA, приём образа по UART |
| Сигналы «готовности» для CI (`CW+READY`, маркеры) | Подпись транзакций по UART (см. §6) |
| Единая дисциплина TX (очередь/мьютекс) | Шифрование трафика UART |

**Версия протокола:** `CWUP/0.1` (строка в ответе `AT+CWINFO?`).

---

## 2. Связь с процессом загрузки и защитой (boot chain)

Текущая схема репозитория CryptoWallet (см. README, `stm32_secure_boot`):

```
[STM32 ROM] → (опционально) [Secure bootloader @ stm32_secure_boot] → [Приложение CryptoWallet @ 0x0800…]
```

- **Защита загрузчика и проверка образа приложения** (подпись прошивки, TrustZone, option bytes) — **вне** CWUP; задаётся проектом `stm32_secure_boot` и линковкой.
- **CWUP действует только после** передачи управления **приложению** (после `main`, инициализации `HW_Init`, старта планировщика FreeRTOS).
- Пока выполняется код загрузчика, по UART может быть **другой** формат (или тишина) — хост **не** интерпретирует это как CWUP до получения преамбулы приложения (§4).

**Итог:** CWUP описывает **диалог с уже запущенным кошельком**, а не с загрузчиком.

---

## 3. Загрузка задач (FreeRTOS) и «готовность»

После `osKernelStart()` живут задачи: Display, Net, Sign, IO, User, (опц.) RNG dump.

**Фазы линии (явно различать в коде, тестах и CI; не смешивать с «сырой» сборкой `USE_RNG_DUMP=1`):**

| Фаза | Смысл | TX от устройства | Примечание |
|------|--------|------------------|------------|
| **A** | Старт до готовности CWUP | Только *TEXT* (`\r\n`); кадры TRNG §7 **не** передаются | Другие задачи могут писать лог на тот же UART (преамбула приложения после загрузчика). Хост **не** считает линию «готовой CWUP», пока не увидит фазу B. |
| **B** | Объявление готовности | Ровно одна строка *TEXT*: `CW+READY,proto=CWUP/0.1,build=<git-describe|unknown>,rng=0|1\r\n` | `rng=1` — если в **этой** сборке предусмотрена выдача TRNG **по CWUP** (`AT+RNG=START`, кадры §7). **Не** означает «сырой» поток `USE_RNG_DUMP`. |
| **C** | Команды, режим TEXT (T) | Ответы `CW+...` на `AT+...` (§5) | До успешного `AT+RNG=START` кадры §7 **не** идут; линия остаётся TEXT. Типичный CI: дождаться `CW+READY` → опционально маркеры → дальнейшие AT. |
| **D** | Сессия бинарного TRNG (T→B по протоколу) | Только кадры §7 на TX | Начинается после `AT+RNG=START` и подтверждения вида `CW+RNG OPEN` (**пока не реализовано** в прошивке). В режиме D на этот UART запрещены `printf`/логи без согласованной блокировки TX. Выход: `AT+RNG=STOP`, таймаут, ошибка RNG, сброс → возврат к TEXT (фаза C по смыслу). |

В прошивке `Core/Src/cwup_uart.c` фазы **A→B→C** отражены состоянием до/после первой строки `CW+READY`; фаза **D** зарезервирована до реализации `AT+RNG=START`.

## 4. Режимы линии (MVP)

| Режим | Код | TX от устройства | RX на устройстве |
|-------|-----|------------------|------------------|
| TEXT | T | Только строки `\r\n`, UTF-8 ASCII | Команды `AT+...` |
| BINARY_RNG | B | Только кадры §7 | Команда `AT+RNG=STOP` (как escape) или таймаут |

Переход **T→B:** только по успешной команде `AT+RNG=START,...` и ответу `CW+RNG OPEN`.

Переход **B→T:** `AT+RNG=STOP`, таймаут сессии, ошибка RNG, или сброс.

**Запрет MVP:** в режиме **B** запрещены `printf` логов на тот же UART (мьютекс TX или глушение логов на время сессии).

---

## 5. Команды (подмножество AT, MVP)

Все команды: строка `AT+<VERB>[=<args>]\r\n`. Ответы: `CW+<CODE>[,<payload>]\r\n` или `CW+ERR=<n>,<msg>\r\n`.

| Команда | Назначение |
|---------|------------|
| `AT+CWINFO?` | Версия протокола, прошивки, флаги сборки |
| `AT+READY?` | Дубль проверки готовности (опционально) |
| `AT+FWINFO?` | **Целостность образа приложения** в Flash: CRC32 (IEEE), длина и границы региона — для сопоставления с `scripts/fw_integrity_check.py` и `build/cryptowallet.bin` (лаборатория / регрессия) |
| `AT+BOOTCHAIN?` | **Текстовое описание цепочки загрузки**, известной приложению: наличие/отсутствие проверенного загрузчика (`stm32_secure_boot`), адрес входа приложения. Не заменяет проверку подписи в ROM-загрузчике |
| `AT+MARKS` | Однократно вывести строки, совместимые с `uart_boot_markers.txt` (для CI без смешения с RNG) |
| `AT+RNG=START,<total_bytes>` | Открыть сессию бинарного потока (кадры §7), не более `total_bytes` полезной нагрузки TRNG |
| `AT+RNG=STOP` | Закрыть сессию, вернуться в TEXT |
| `AT+PING` | Проверка связи: `CW+PONG` |
| `AT+WALLET?` | **Лаборатория / CI (без аутентификации):** доступность seed через `get_wallet_seed()` (0/1) и флаг сборки `USE_CRYPTO_SIGN` (0/1). **Не** выдаёт ключи и **не** заменяет защищённый канал |
| `AT+SELFTEST?` | Минимальная проверка: `CW+SELFTEST,ok=1,tick=<ticks>` (FreeRTOS tick) |
| `AT+ECHO=<text>` | Возврат печатного ASCII в `CW+ECHO,<text>` (до 120 символов; непечатаемые символы отбрасываются). Для регрессии конвейера host↔UART |

**Ответы (пример):**

- `AT+FWINFO?` → `CW+FWINFO,FWINFO crc32=0x12345678,len=...,start=0x0800....,end=0x0800....` (строка совпадает по смыслу с логом при старте и с `fw_integrity_snprint()` в `fw_integrity.c`).
- `AT+BOOTCHAIN?` → `CW+BOOTCHAIN,rom=STM32,verified_boot=0|1,note=...` (детали зависят от сборки и макросов; при отсутствии данных — `CW+ERR=12,bootchain unknown`).
- `AT+WALLET?` → `CW+WALLET,seed=0|1,crypto_sign=0|1` (`seed=1` если `get_wallet_seed()` успешен, напр. `USE_TEST_SEED=1` в dev).
- `AT+SELFTEST?` → `CW+SELFTEST,ok=1,tick=…`
- `AT+ECHO=hi` → `CW+ECHO,hi`

**Ошибки:** `CW+ERR=1,unknown command`, `CW+ERR=10,rng fault`, `CW+ERR=11,busy`, …

## 6. Операции кошелька и каналы (полный набор — матрица)

В MVP **операции кошелька** (баланс логики приложения) остаются на **существующих каналах**; UART **не** дублирует подпись для продукта.

| Операция | Канал в продукте | CWUP-0.1 |
|----------|------------------|----------|
| Проверка живости / диагностика | UART (опц.), OLED | `AT+PING`, `AT+CWINFO?`, `CW+READY` |
| Захват TRNG для тестов | UART | `AT+RNG=START` / кадры |
| Проверка загрузки (маркеры) | UART | `AT+MARKS` или legacy-лог |
| Проверка целостности прошивки в рантайме | UART | `AT+FWINFO?` + хост `fw_integrity_check.py` |
| Диагностика цепочки доверия (лаб.) | UART | `AT+BOOTCHAIN?` + документация `stm32_secure_boot` |
| Запрос статуса сети / кошелька | HTTP | как сейчас (`task_net`) |
| Отправка TX на подпись | HTTP/WebUSB | как сейчас |
| Подтверждение пользователем | Кнопка, UI | не через UART MVP |
| Обновление прошивки | Bootloader / ST-Link | **вне CWUP** |

**Расширение после MVP:** опциональные команды `AT+WALLET=...` (только lab, с риском утечки метаданных) — **не** входят в CWUP-0.1.

## 7. Кадр бинарного TRNG (MVP)

Формат фиксированной длины (пример 32 байта полезной нагрузки на кадр):

```
[0..1]  magic = 0xC7 0x57   ('ÇW')
[2..3]  u16 payload_len = N  (LE, N ≤ 256)
[4..]   N байт TRNG
[4+N .. 4+N+3] u32 CRC32 (IEEE) по байтам [0 .. 4+N-1] (без CRC)
```

Хост **склеивает** только байты TRNG из поля полезной нагрузки; CRC проверяет целостность кадра. При ошибке CRC — пропуск кадра + счётчик в лог CI.

## 8. Таймауты (рекомендуемые)

| Параметр | Значение |
|----------|----------|
| Ожидание первой команды после `CW+READY` | 120 s (как CI_UART) |
| Сессия `AT+RNG=START` без прогресса | 180 s stall |
| Межкадровый интервал | не нормируется в MVP (зависит от задачи RNG) |

## 10. Статус реализации (прошивка)

Условие текстового CWUP: **`USE_RNG_DUMP` не задан** при сборке. Приём: USART3 (`HAL_UART_Receive_IT`), строки до `\r\n`, задача **`CWUP`**, TX через общий мьютекс с логами (`UART_Tx_Lock`). При **`USE_RNG_DUMP=1`** на том же UART — сырой бинарный TRNG, **`Cwup_Init()`** не поднимает CWUP (см. также §11).

| Команда / элемент | Статус | Примечание |
|-------------------|--------|------------|
| Строка **`CW+READY,...`** при старте | Реализовано | Фаза B |
| **`AT+PING`** | Реализовано | `CW+PONG` |
| **`AT+CWINFO?`** | Реализовано | протокол, build |
| **`AT+READY?`** | Реализовано | дубль `CW+READY` |
| **`AT+FWINFO?`** | Реализовано | `fw_integrity_snprint()` |
| **`AT+BOOTCHAIN?`** | Реализовано | лаб., макросы + `VTOR` |
| **`AT+WALLET?`** | Реализовано | lab / CI, без секретов |
| **`AT+SELFTEST?`** | Реализовано | tick FreeRTOS |
| **`AT+ECHO=`** | Реализовано | регрессия конвейера |
| **`AT+MARKS`** | Не реализовано | только спецификация §5 |
| **`AT+RNG=START`**, **`AT+RNG=STOP`**, кадры §7 | Не реализовано | фаза D; не путать с «сырым» `USE_RNG_DUMP` |

## 11. Нерешённые вопросы (v0.2+)

- Реализация **`AT+MARKS`** и **кадрированного TRNG** (§7, `AT+RNG=START`) — см. таблицу §10.
- Согласование с USB CDC, если UART занят.
- Совместимость с `USE_RNG_DUMP` «сырой поток» — идея `CWUP_LEGACY_RAW=1` в сборке (не в коде).
