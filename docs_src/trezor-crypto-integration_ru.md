# Интеграция trezor-crypto

<brief>CryptoWallet интегрирует trezor-crypto для Bitcoin криптографических операций: генерация BIP-39 мнемоники, BIP-32 HD вывод ключей, подпись ECDSA secp256k1 с RFC6979, хеширование SHA-256 и управление энтропией через STM32 TRNG + смешивание пула энтропии.</brief>

## Обзор

Прошивка CryptoWallet делегирует все криптографические операции **trezor-crypto**, well-tested, production-grade криптографической библиотеке (MIT License), используемой в аппаратных кошельках Trezor. Интеграция условна от флага компиляции `USE_CRYPTO_SIGN=1`; без него доступна только SHA-256 через `sha256_minimal.c`.

## Криптографические компоненты

### 1. Генерация BIP-39 мнемоники

**Модуль:** `bip39.c`

**Цель:** Преобразовать 128-битную энтропию в 12-словную мнемоническую фразу, используя BIP-39 английский словарь.

**Реализация в CryptoWallet:**
```c
int crypto_entropy_to_mnemonic_12(const uint8_t entropy[16],
                                  char *mnemonic_out, size_t mnemonic_size);
```

**Поток:**
1. Генерировать 128-битную энтропию из `random_buffer()` (STM32 TRNG + пул)
2. Вызвать trezor's `mnemonic_from_data(entropy, 16, mnemonic_out)`
3. Вывод: разделённые пробелами 12 слов (например, "abandon wallet ... about")

**Пример тестового вектора:**
- Энтропия: все нули (не рекомендуется в production)
- Мнемоника: "abandon abandon abandon ... abandon about"
- Bitcoin адрес: `1LqBGSKuX5yYUonjxT5qGfpUsXKYYWeabA`

### 2. BIP-39 мнемоника → Семя

**Модули:** `pbkdf2.c`, `hmac.c`

**Цель:** Преобразовать мнемонику + опциональную парольную фразу в 64-байтное семя для BIP-32 вывода.

**Реализация:**
```c
// Обёрнуто в trezor; CryptoWallet вызывает:
int mnemonic_to_seed(const char *mnemonic, const char *passphrase,
                     uint8_t seed_out[64], size_t max_len);
```

**Параметры PBKDF2 (стандарт BIP-39):**
- HMAC-SHA512
- Итерации: 2048
- Соль: префикс "TREZOR" + парольная фраза

**Вывод:** 64 байта для использования как корневое семя для BIP-32.

### 3. BIP-32 иерархические детерминированные ключи

**Модули:** `bip32.c`, `curves.c`, `secp256k1.c`

**Цель:** Вывести дочерние ключи из мастер-семени вдоль определённого пути вывода.

**Стандартный путь в CryptoWallet:** `m/44'/0'/0'/0/0`

- `44` — BIP-44 (multi-purpose HD кошельки)
- `0'` — Bitcoin монета (усиленная)
- `0'` — Счёт 0 (усиленный)
- `0` — Внешняя цепь
- `0` — Первый адрес

**Реализация в CryptoWallet:**
```c
int crypto_derive_btc_m44_0_0_0_0(const uint8_t *seed, size_t seed_len,
                                  uint8_t priv_key_out[32]);
```

**Процесс вывода:**
1. `hdnode_from_seed()` — Инициализировать корневой HDNode из 64-байтного семени
2. `hdnode_private_ckd()` — Вывести на каждом усиленном/неусиленном шаге
3. `hdnode_fill_public_key()` — Вычислить открытый ключ (опционально, для проверки)
4. Вывод: 32-байтный приватный ключ (big-endian скаляр)

**Безопасность:** Усиленный вывод (`'` нотация) обеспечивает, что дочерние ключи не могут быть выведены только из открытого ключа.

### 4. Подпись ECDSA (secp256k1)

**Модули:** `ecdsa.c`, `secp256k1.c`, `bignum.c`, `rfc6979.c`

**Цель:** Подписать дайджест сообщения, используя ECDSA с эллиптической кривой secp256k1 (стандарт Bitcoin).

**Реализация в CryptoWallet:**
```c
int crypto_sign_btc_hash(uint8_t priv_key[32],
                        const uint8_t hash[32],
                        uint8_t sig_out[64]);
```

**Алгоритм подписи:**
- **Кривая:** secp256k1 (p = 2^256 - 2^32 - 977)
- **Алгоритм хеша:** SHA-256
- **RFC6979:** Детерминированная генерация k (без зависимости RNG)
- **Вывод:** 64-байтная компактная подпись (r || s, оба 32-байтных big-endian)

**Детерминированный k (преимущества RFC6979):**
- Один и тот же приватный ключ + одно и то же сообщение = идентичная подпись
- Предотвращает атаки на отказ RNG
- Позволяет проверку подписи в тестах

### 5. Хеширование SHA-256

**Модули:** `sha2.c`, `hasher.c`

**Цель:** Хешировать произвольные данные, используя SHA-256 (FIPS 180-4).

**Реализация в CryptoWallet:**
```c
int crypto_hash_sha256(const uint8_t *data, size_t len,
                       uint8_t digest_out[32]);
```

**Входные данные:** Каноническая строка транзакции, отформатированная как `"recipient|amount|currency"`

**Пример:**
- Ввод: `"1LqBGSKuX5yYUonjxT5qGfpUsXKYYWeabA|0.5|BTC"`
- Вывод: 32-байтный SHA-256 дайджест

**Fallback:** Когда `USE_CRYPTO_SIGN=0`, делегирует на `sha256_minimal.c`.

### 6. Проверка адреса (Base58Check)

**Модули:** `base58.c`, `address.c`

**Цель:** Проверить Bitcoin адреса, используя кодирование Base58Check (проверка контрольной суммы).

**Использование в CryptoWallet:** `tx_request_validate.c` вызывает trezor's `base58_decode_check()`.

**Поддерживаемый формат:** Старые P2PKH адреса (начинаются с '1', 25 байт всего).

### 7. Энтропия и генерация случайных чисел

**Модули:** `rand.c`, `hmac_drbg.c`

**Интеграция STM32H743 TRNG:**
```c
int crypto_rng_init(void);  // Вызывается один раз из main.c
void random_buffer(uint8_t *buf, size_t len);  // Вызывается trezor-crypto
```

**Источники энтропии:**
1. **Аппаратный TRNG:** `HAL_RNG_GenerateRandomNumber()` → 32-битные слова
2. **Пул энтропии:** Смешивание программного LCG (XOR + обновление состояния)
3. **Комбинация:** `rng_val = HAL_RNG_value XOR entropy_pool`

**Параметры LCG (Numerical Recipes):**
- Множитель: `1664525`
- Приращение: `1013904223`
- Обновление состояния: `pool = (pool * 1664525) + 1013904223`

**Качество:** Комбинирует аппаратную случайность с программным отбеливанием, чтобы защиться от смещения TRNG.

## Полный конвейер подписи

```
Сетевой ввод (HTTP POST /tx или WebUSB)
    ↓
[tx_request_validate.c]
    ├─ Base58Check декодирование: адрес получателя
    ├─ Regex: формат суммы (десятичный)
    ├─ Whitelist: валюта ("BTC", "ETH", ...)
    ↓
[task_sign.c → crypto_wallet.c]
    ├─ build_hash_input(): Форматировать "recipient|amount|currency"
    ├─ crypto_hash_sha256(): 32-байтный дайджест
    ├─ Подтверждение пользователя: g_user_event_group (нажатие кнопки)
    ├─ get_wallet_seed(): 64-байтное BIP-39 семя
    ├─ crypto_derive_btc_m44_0_0_0_0(): 32-байтный приватный ключ
    ├─ crypto_sign_btc_hash(): 64-байтная подпись (r||s)
    ├─ memzero(): Очистить конфиденциальные буферы
    ↓
[Вывод]
    ├─ USB: WebUSB_NotifySignatureReady(64-байтная сигнатура)
    ├─ Сеть: HTTP JSON ответ с hex-кодированной подписью
    ↓
Завершено
```

## Опции компиляции

### `USE_CRYPTO_SIGN=1` (По умолчанию для production)

**Связанные модули (22 всего):**
- **BIP:** bip39, bip32
- **ECDSA:** ecdsa, secp256k1, curves, bignum, rfc6979
- **Hash:** sha2, hasher, sha3, blake256, blake2b, blake2s, groestl
- **HMAC:** hmac, hmac_drbg, pbkdf2
- **Кодирование:** base58, address, ripemd160
- **Прочее:** rand, nist256p1
- **ED25519:** (связано, но не используется в текущем CryptoWallet)

**Размер бинара:** ~100-150 KB объектного кода

**Особенности:**
- Полная подпись ECDSA
- Вывод BIP-39/BIP-32
- STM32 TRNG + пул энтропии
- Детерминированный k RFC6979

### `USE_CRYPTO_SIGN=0` (Минимальная сборка)

**Связанные модули:**
- `sha2.c` (только SHA-256 из trezor-crypto)
- `sha256_minimal.c` (fallback, ~16 KB)

**Размер бинара:** ~5 KB

**Особенности:**
- Только SHA-256 хеширование
- Нет ECDSA, BIP-39, BIP-32
- Нет вывода ключей

**Случай использования:** Тестирование, проверка UI, минимальные встроенные системы.

### `USE_TEST_SEED=1` (Только разработка)

**Поведение:**
- `get_wallet_seed()` возвращает hardcoded BIP-39 тестовый вектор
- Мнемоника: "abandon abandon ... about"
- Выведенный Bitcoin адрес: `1LqBGSKuX5yYUonjxT5qGfpUsXKYYWeabA`
- Позволяет воспроизводимые подписи для автоматических тестов

**⚠️ ПРЕДУПРЕЖДЕНИЕ:** Никогда не используйте в production или с реальными средствами.

## Соображения безопасности

### 1. Качество энтропии

**Многоисточниковое смешивание:**
- STM32H743 аппаратный TRNG обеспечивает физическую случайность
- Пул энтропии программного обеспечения добавляет временную дисперсию
- Комбинированно через XOR, чтобы защиться от отказа одного источника

**Проверка:**
- Скрипты: `scripts/capture_rng_uart.py`, `scripts/run_dieharder.py`
- Статистические тесты NIST FIPS 140-2 рекомендуются

### 2. Обработка материала ключей

**Обнуление:**
- Все конфиденциальные буферы (семя, приватный ключ, дайджест) очищаются через `memzero()`
- Использует волатильные записи, чтобы предотвратить оптимизацию компилятора
- Ключи никогда не логируются, не отображаются или не передаются незашифрованными

**Время жизни буфера:**
- Приватные ключи выделены в стеке (локальная область видимости)
- Очищены перед возвратом функции
- Нет глобального хранилища ключей в прошивке

### 3. Детерминизм подписи

**Преимущество RFC6979:**
- Удаляет зависимость RNG из генерации ECDSA `k`
- Предотвращает случайное переиспользование ключа (один и тот же хеш → одна и та же подпись)
- Поддерживает детерминированные тестовые векторы

**Trade-off:** Подпись детерминирована, не случайна (стандарт в Bitcoin).

### 4. Безопасность мнемоники

**Защита:**
- Семя BIP-39 вычисляется по требованию, никогда не сохраняется
- Сразу же обнулено после вывода ключа
- Список мнемоник не сохранён в памяти flash прошивки

**Production реализация:**
- Развернуть безопасный элемент (SE) или зашифрованную flash для хранилища семян
- Использовать KDF с дополнительной парольной фразой пользователя
- Рассмотреть интеграцию аппаратного кошелька
