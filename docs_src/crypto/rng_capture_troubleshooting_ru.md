# Руководство по Устранению Неполадок Захвата RNG

## Обнаруженная Ошибка

```
✗ Failed to capture RNG: device reports readiness to read but returned no data 
  (device disconnected or multiple access on port?)
```

Эта ошибка возникает когда последовательный порт открывается успешно, но возвращает **нет данных**. Это обычно означает:

1. **Устройство не подключено или отключено**
2. **Несколько программ обращаются к одному порту** (что-то уже читает)
3. **Прошивка не построена с `USE_RNG_DUMP=1`**
4. **Устройство не выводит данные RNG на UART**

---

## Пошаговое Устранение Неполадок

### 1. Проверьте Подключение Устройства

Сначала проверьте физическое подключение устройства:

```bash
# Список всех устройств USB/Serial
ls -la /dev/tty* | grep -E "(USB|ACM|FTDI|Serial)"

# Или используйте lsusb чтобы увидеть USB-устройства
lsusb | grep -i stm32

# Или проверьте dmesg для последних подключений
dmesg | tail -20 | grep -E "usb|tty"
```

**Ожидаемый вывод**: Должно показать `/dev/ttyACM0`, `/dev/ttyUSB0` или похожее

**Если устройство не появляется**:
- Проверьте подключение кабеля USB к плате STM32H743
- Попробуйте другой порт USB
- Попробуйте другой кабель USB
- Перезагрузите устройство (отключение/включение питания)

---

### 2. Проверьте Разрешения Порта

Проверьте есть ли у вас разрешение на доступ к последовательному порту:

```bash
# Проверьте текущие разрешения
ls -la /dev/ttyACM0

# Вы должны увидеть что-то вроде:
# crw-rw---- 1 root dialout 166, 0 ...

# Если вас нет в группе dialout:
sudo usermod -a -G dialout $USER
newgrp dialout

# Выйдите и снова войдите, затем проверьте:
id | grep dialout
```

**При отказе в доступе**:
```bash
# Временное решение (текущая сессия)
sudo chmod 666 /dev/ttyACM0

# Постоянное решение (добавить в группу dialout - смотрите выше)
```

---

### 3. Проверьте Конфликты Портов

Другая программа может использовать порт (монитор серийный, отладчик IDE и т.д.):

```bash
# Посмотрите что использует порт
sudo lsof /dev/ttyACM0

# Или проверьте заблокирован ли порт процессом
fuser /dev/ttyACM0

# Типичные подозреваемые для закрытия:
# - Arduino IDE Serial Monitor
# - VSCode/Cursor Serial Monitor
# - miniterm/picoterm
# - Другие экземпляры этого скрипта
```

**Если порт в использовании**:
- Закройте все мониторы последовательных портов IDE
- Завершите другие экземпляры теста: `pkill -f test_rng_signing_comprehensive.py`
- Подождите 2-3 секунды перед повторной попыткой

---

### 4. Проверьте Конфигурацию Прошивки

**Критично**: Прошивка **должна быть построена с `USE_RNG_DUMP=1`**

Это настраивает STM32 на вывод **сырых данных RNG** на UART (только двоичный, без текста).

#### Проверьте Makefile или Конфигурацию Сборки

```bash
# Найдите параметр USE_RNG_DUMP
grep -r "USE_RNG_DUMP" /data/projects/CryptoWallet/

# Должно найти что-то вроде:
# - Makefile: USE_RNG_DUMP=1
# - CMakeLists.txt или конфигурация сборки
# - Или установлено в Core/Src/main.h
```

#### Если НЕ Установлено

Вам нужно **пересобрать прошивку** с этим флагом:

```bash
cd /data/projects/CryptoWallet

# Вариант 1: Установите в Makefile
sed -i 's/USE_RNG_DUMP=0/USE_RNG_DUMP=1/' Makefile

# Вариант 2: Установите как переменную окружения
export USE_RNG_DUMP=1

# Затем пересоберите
make clean
make -j$(nproc)

# Загрузите на устройство (если ваш проект имеет целевой флеш)
make flash
```

#### Проверьте Прошивку

После загрузки протестируйте UART вручную:

```bash
# Установите miniterm или похожее
pip install pyserial

# Читайте сырые данные UART (остановка Ctrl+C)
python3 -m serial.tools.miniterm /dev/ttyACM0 115200 --raw

# Вы должны увидеть двоичный мусор (данные RNG)
# НЕ: текстовый вывод, приглашения меню или ничего
```

---

### 5. Протестируйте Коммуникацию UART Вручную

Перед запуском полного теста проверьте базовую последовательную коммуникацию:

```python
#!/usr/bin/env python3
# Простой тест серийных портов
import serial
import time

port = "/dev/ttyACM0"
try:
    ser = serial.Serial(port=port, baudrate=115200, timeout=2)
    time.sleep(0.5)
    
    print(f"Порт {port} открыт успешно")
    print(f"В ожидании: {ser.in_waiting}")
    
    # Попробуйте прочитать некоторые данные
    data = ser.read(100)
    print(f"Прочитано {len(data)} байтов")
    print(f"Первые 16 байтов (hex): {data[:16].hex()}")
    
    ser.close()
    print("✓ Коммуникация по серийному порту работает")
except Exception as e:
    print(f"✗ Ошибка: {e}")
```

Сохраните как `test_serial.py` и запустите:
```bash
python3 test_serial.py
```

**Ожидаемое**:
- Порт открывается успешно
- `in_waiting` показывает доступные данные
- Читает 100+ байтов
- Первые байты двоичные (не ASCII текст)

## Быстрый Диагностический Контрольный Список

Запустите эти проверки по порядку:

```bash
# 1. Устройство подключено?
ls /dev/ttyACM0
# Ожидаемое: /dev/ttyACM0 (или /dev/ttyUSB0)

# 2. Можете читать с порта?
sudo dd if=/dev/ttyACM0 bs=1 count=10 2>/dev/null | xxd
# Ожидаемое: Несколько байтов (двоичные данные)

# 3. Python может его открыть?
python3 -c "import serial; s=serial.Serial('/dev/ttyACM0'); print('✓')"
# Ожидаемое: ✓ и никакой ошибки

# 4. Прошивка имеет USE_RNG_DUMP=1?
grep -i "USE_RNG_DUMP" /data/projects/CryptoWallet/Makefile
# Ожидаемое: USE_RNG_DUMP=1

# 5. Прошивка недавно загружена?
ls -la /data/projects/CryptoWallet/build/*.elf
# Если старше, чем ожидалось, нужна пересборка и загрузка
```

## Типичные Проблемы и Решения

| Проблема | Симптомы | Решение |
|---------|----------|--------|
| **Устройство не подключено** | "No such file or directory: /dev/ttyACM0" | Проверьте кабель USB и порт |
| **Множественный доступ** | "Failed to capture RNG: no data returned" | Закройте мониторы портов, завершите другие экземпляры |
| **Проблема прошивки** | "No data returned" (но порт открывается) | Пересоберите с `USE_RNG_DUMP=1` и загрузите |
| **Несоответствие скорости передачи** | "Garbage or no data" | Проверьте скорость передачи (115200) |
| **Отказ в доступе** | "Permission denied" | Добавьте пользователя в группу dialout |
| **Другой путь устройства** | Работает для других, но не для вас | Используйте `--port /dev/ttyUSB0` или проверьте `ls /dev/tty*` |

## Если Всё Ещё Не Работает

Попробуйте эти дополнительные шаги:

### Сброс Устройства
```bash
# Цикл питания (отключите и переподключите USB)
# Или используйте сброс ST-Link если доступен

# Затем подождите 2-3 секунды и попробуйте снова
python3 scripts/test_rng_signing_comprehensive.py --mode rng
```

### Пересоберите Всё
```bash
cd /data/projects/CryptoWallet

# Полная очистка
make distclean 2>/dev/null || true

# Пересоберите с явным RNG dump
make USE_RNG_DUMP=1 -j$(nproc)

# Загрузите (зависит от вашей конфигурации)
make flash  # или похожее для вашей конфигурации
```

### Тестируйте Другим Инструментом
```bash
# Попробуйте альтернативные инструменты для проверки серийных портов

# Вариант 1: miniterm (простой)
python3 -m serial.tools.miniterm /dev/ttyACM0 115200 --raw

# Вариант 2: screen
screen /dev/ttyACM0 115200

# Вариант 3: picocom
sudo apt install picocom
picocom -b 115200 /dev/ttyACM0
```

Если они показывают двоичные данные, устройство и порт работают. Тогда проблема в конфигурации прошивки.

### Проверьте Логи
```bash
# Включите вывод отладки в тестовом скрипте
python3 scripts/test_rng_signing_comprehensive.py --mode rng -v 2>&1 | tee test_debug.log

# Просмотрите лог для деталей
cat test_debug.log | grep -i "error\|failed\|timeout"
```

## Следующие Шаги

1. **Запустите диагностический контрольный список выше** - определите какой шаг не прошел
2. **Исправьте конкретную проблему** - используйте решение из таблицы
3. **Повторите попытку теста**:
   ```bash
   source .venv-test/bin/activate
   python3 scripts/test_rng_signing_comprehensive.py --mode rng
   ```

4. **Если всё ещё не работает** - проверьте:
   - Лог устройства: `dmesg | tail -20`
   - Исходник прошивки: `/data/projects/CryptoWallet/Core/Src/` для конфигурации RNG
   - Вывод сборки: Проверьте вывод `make` для предупреждений/ошибок
