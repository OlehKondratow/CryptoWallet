# Улучшения безопасности и качества TRNG (предложения)

Документ фиксирует рекомендации относительно текущей схемы: `HAL_RNG` в `crypto_wallet.c`, сырой дамп в `rng_dump.c`, инициализация в `hw_init.c` (`MX_RNG_Init`).

Приоритеты: **P0** — критично для продукта, **P1** — сильно желательно, **P2** — усиление и соответствие best practice.

---

## P0 — Критично

### 1. Единый и корректный доступ к аппаратному RNG

- **`rng_dump.c`** сейчас читает **`RNG->DR` в цикле** без проверки **`RNG_SR`** (готовность данных, флаги ошибок тактирования/seed). В RM STM32H7 между опросами действуют ограничения; при ошибке RNG выдача может быть недопустимой.
- **`crypto_wallet.c`** использует **`HAL_RNG_GenerateRandomNumber`** — предпочтительный путь.
- **Рекомендация:** везде использовать HAL (или обёртку), проверять **`HAL_OK`**, при ошибке — **не** подставлять предсказуемый LCG-поток для криптографии (см. п.2).

### 2. Поведение при отказе TRNG (fail-closed)

- В **`random_buffer()`** при неуспехе HAL выполняется только обновление **LCG** (`1664525U * pool + …`) — это **не** криптостойкий генератор.
- **Рекомендация:** для операций с ключами/подписями при длительной недоступности TRNG: **остановка крипто-операций**, явная ошибка, лог (без утечки секретов), опционально **безопасный сброс**. Не генерировать ключи и seed из одного LCG.

### 3. Постобработка (conditioning) по NIST SP 800-90C

- Сырые выходы TRNG обычно **подмешивают/вытягивают** через **Hash_DRBG / HMAC_DRBG / ChaCha20 DRBG**, инициализированный **достаточным** объёмом энтропии с HW RNG.
- **Рекомендация:** как в Trezor-подобных стеках — **DRBG поверх TRNG** для всего, что идёт в `random_buffer` для кошелька; XOR с LCG заменить на **явное смешивание** (например SHA-256 от `(prev_state || trng_word || counter)` или готовый лёгкий DRBG).

## P1 — Сильно желательно

### 4. Непрерывные проверки здоровья (health tests)

- Включить и **обрабатывать** события: **clock error**, **seed error** (STM32 RNG), счётчики повторов (repetition count) / упрощённые **adaptive** проверки по SP 800-90B там, где это реализуемо на МК.
- При срабатывании — тот же **fail-closed**, что в п.2.

### 5. Усилить `crypto_rng_init()`

- Сейчас: **`HAL_GetTick()`** + побитовые сдвиги — мало энтропии.
- **Рекомендация:** смешать в пул (не как единственный источник случайности, а как **доп. размагничивание**): **UID MCU**, **значения reset flags**, при наличии — **ADC noise** (несколько выборок неиспользуемого канала), **время старта** с высокоразрешающего таймера. Всё это только **дополнение** к TRNG, не замена.

### 6. Изолировать режим `USE_RNG_DUMP`

- Убедиться, что в **production-сборке** дамп RNG на UART **невозможен** (отдельный флаг, отсутствие в release, защита от включения по ошибке).

### 7. Тестирование и регрессия

- Автоматизировать: минимальный размер захвата + **dieharder** / **nist_sts** на CI при **`CI_BUILD_USE_RNG_DUMP=1`** (отдельный workflow или ночной прогон).
- Вести **версию прошивки** в отчёте рядом с результатами RNG.

## Краткая таблица «сейчас → лучше»

| Область | Сейчас (риск) | Направление |
|--------|----------------|------------|
| Доступ к DR | Прямое чтение `RNG->DR` в дампе | HAL + проверка статуса |
| Fallback | LCG при отказе HAL | Fail-closed или DRBG только от успешного TRNG |
| Смешивание | XOR с LCG | Hash/DRBG (SP 800-90C) |
| Init пула | В основном `GetTick` | + UID, reset flags, опц. ADC |
| Режим дампа | Сырой поток наружу | Только диагностика, не prod |
