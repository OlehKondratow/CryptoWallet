# `main.c` + `main.h`

<brief>Модуль `main` — это системный оркестратор приложения: инициализирует железо (HAL, ранняя инициализация LwIP, тактирование), устанавливает контракты IPC (очереди, семафоры, группы событий, глобальное состояние), инициализирует сервисы времени/криптографии, и запускает семейство основных задач FreeRTOS (дисплей, сеть, подпись, IO, ввод пользователя). Точка входа для всего встроенного приложения кошелька.</brief>

## Обзор

`main.c` — это "код связи", который превращает независимые блоки железа и модули приложения в работающую систему. Это **не** конечный автомат и **не** постоянный рабочий процесс — это **одноразовая последовательность загрузки**, которая:

1. Инициализирует железо в правильном порядке (критично для LwIP/DMA/когерентности кеша)
2. Создаёт примитивы межпроцессной коммуникации (IPC), на которые будут опираться все задачи
3. Инстанцирует семейство задач и запускает планировщик FreeRTOS
4. Никогда не возвращает управление (планировщик работает бесконечно)

## Абстракция (синтез логики)

**Проблема:** встроенному кошельку нужно:
- Принимать запросы на подпись из сети (HTTP) или USB-интерфейсов
- Позволять пользователю подтвердить/отклонить через кнопку
- Отображать статус на OLED 128×32
- Выполнять подпись ECDSA при включенной криптографии
- Безопасно управлять несколькими одновременными задачами (без race condition)

**Решение:** `main.c` предоставляет **архитектурный контракт**:

1. **Порядок инициализации железа важен:** LwIP требует, чтобы MPU/кеш были настроены **до** `HAL_Init()`. Это неочевидно, но воспроизводится из примера `lwip_zero` в STM32CubeH7.

2. **Глобальное состояние намеренно:** все задачи совместно используют:
   - `g_tx_queue`: запросы транзакций из сетевого слоя в слой подписания
   - `g_display_queue`: данные транзакции (для дисплея) от сети
   - `g_user_event_group`: события кнопки сигнализируют подтверждение/отклонение
   - `g_i2c_mutex`, `g_ui_mutex`, `g_display_ctx_mutex`: сериализуют доступ к шине I2C (OLED) и состоянию UI

3. **Порядок создания задач детерминирован:** задача дисплея создаётся первой (приоритет логирования), затем сеть, затем подпись, затем индикаторы и ввод пользователя. Это обеспечивает ранний диагностический вывод.

4. **Хуки ловят катастрофические отказы:**
   - `vApplicationMallocFailedHook()`: нет доступной памяти
   - `vApplicationStackOverflowHook()`: переполнение стека задачи
   - `Error_Handler()`: финальная точка выхода, которая логирует и останавливает систему

### Фаза 1: выполнение задач (непрерывная)

Когда `osKernelStart()` вызывается, планировщик запускает задачи с вытеснением:

- **Task_Display**: читает события, обновляет SSD1306 ~1 Гц
- **Task_Net**: слушает Ethernet/HTTP, ставит транзакции в очередь
- **Task_Sign**: ожидает транзакции → показывает на дисплее → ожидает кнопки → подписывает
- **Task_IO**: мигает LED1 (жива), устанавливает LED2 (статус сети), LED3 (тревога)
- **Task_User**: дебаунс кнопки → отправляет подтверждение/отклонение на подпись

Весь общий доступ к данным защищён мьютексами; очереди и группы событий гарантируют безопасный переход между потоками.

## Регистры и периферия

### Блок управления системой (SCB) — Cortex-M7

| Регистр/поле | Операция | Причина |
|---|---|---|
| `SCB->CCR` | Очистить `SCB_CCR_UNALIGN_TRP_Msk` | Стек LwIP может получать доступ к неправильно выровненным полям в заголовках протоколов (TCP/IP). Без этого M7 выбросит hard fault. |

**Почему?** Процессор Cortex-M7 может выбросить ошибку при неправильно выровненных load/store операциях. Но LwIP использует C struct, которые могут быть не выровнены по слову в памяти. Другие реализации (newlib, lwip_zero) также отключают эту ловушку.

### Порядок инициализации периферии

| Периферия | Модуль | Время | Причина |
|---|---|---|---|
| MPU | `HW_Init_Early_LwIP()` | До HAL_Init | Когерентность кеша/DMA для буферов LwIP |
| Тактирование | `HAL_Init()` | Рано | Настройка PLL, тики прерываний |
| GPIO | `HW_Init()` | После HAL | LED, кнопка, AF настройка |
| I2C1 | `HW_Init()` | После GPIO | Инициализация SSD1306 (зависит от GPIO AF) |
| UART3 | `HW_Init()` | После GPIO | Бэкенд логирования |
| USB | `HW_Init()` | После GPIO | Инициализация WebUSB device |
| RNG | `HW_Init()` + `crypto_rng_init()` | После тактирования | Посевный энтропии |
| SSD1306 | `HW_Init()` → `ssd1306.c` | После I2C | Дисплей готов до задач |

## Время и условия ветвления

### Флаги сборки

| Флаг | Тип | Поведение |
|---|---|---|
| `USE_LWIP` | Время компиляции | Включает вызов `HW_Init_Early_LwIP()`. Без него, настройка MPU/кеша пропускается. |
| `BOOT_TEST` | Время компиляции | **Диагностический режим:** пропускает FreeRTOS, цикл с `HAL_Delay(500)` и переключением LED. Полезно для тестирования HAL без планировщика задач. |
| `SKIP_OLED` | Время компиляции (по умолчанию 0) | Если установлено в 1, пропускает инициализацию SSD1306 и вывод баннера. |
| `USE_CRYPTO_SIGN` | Время компиляции | Если установлено, вызывает `crypto_rng_init()` для посева пула энтропии trezor-crypto. |

### Условия ошибок

| Условие | Обработчик | Результат |
|---|---|---|
| Ошибка при создании очереди/семафора (NULL check) | `Error_Handler()` | Логирует `[ERR]`, бесконечный цикл; система остановлена. |
| Память в кучи FreeRTOS исчерпана | `vApplicationMallocFailedHook()` | Логирует `[MALLOC FAIL]`, бесконечный цикл. |
| Обнаружено переполнение стека задачи | `vApplicationStackOverflowHook()` | Логирует `[STACK OVF имя_задачи]`, бесконечный цикл. |
| `osKernelStart()` неудачен | `Error_Handler()` | Не должно быть; указывает на ошибку конфигурации FreeRTOS. |

## Глобальное состояние, инициализированное в `main()`

### Очереди

```c
g_tx_queue = xQueueCreate(4U, sizeof(wallet_tx_t));
```
- **Размер:** 4 элемента
- **Элемент:** `wallet_tx_t` (запрос транзакции: получатель, сумма, валюта)
- **Производитель:** `task_net.c` (обработчик HTTP POST `/tx`)
- **Потребитель:** `task_sign.c` (FSM подписания)
- **Цель:** отделяет сетевой поток от FSM подписания

```c
g_display_queue = xQueueCreate(4U, sizeof(Transaction_Data_t));
```
- **Размер:** 4 элемента
- **Элемент:** `Transaction_Data_t` (подмножество tx для дисплея)
- **Производитель:** `task_net.c` (копирует из входящего запроса)
- **Потребитель:** `task_display.c` (отображает на SSD1306)
- **Цель:** дисплей видит транзакцию сразу; FSM подписания может всё ещё обрабатывать

### Группа событий

```c
g_user_event_group = xEventGroupCreate();
```
- **Используемые биты:**
  - `USER_CONFIRM_BIT` (бит 0): пользователь нажал кнопку (< 2.5 с)
  - `USER_REJECT_BIT` (бит 1): пользователь удерживал кнопку (≥ 2.5 с)
- **Производитель:** `task_user.c` (логика дебаунса + расчёт времени)
- **Потребитель:** `task_sign.c` (ожидание битов события через `xEventGroupWaitBits()`)
- **Цель:** неблокирующий сигнал от обработчика кнопки

### Мьютексы

```c
g_i2c_mutex = xSemaphoreCreateMutex();    // шина I2C1
g_ui_mutex = xSemaphoreCreateMutex();      // буфер UI текста
g_display_ctx_mutex = xSemaphoreCreateMutex(); // display_context_t
```

| Мьютекс | Защищает | Задачи | Таймаут |
|---|---|---|---|
| `g_i2c_mutex` | Железо I2C1 (GPIO AF, DR, статус) | task_display, task_io | Бесконечный (нет риска инверсии приоритета) |
| `g_ui_mutex` | Отформатированный буфер текста для дисплея | task_display, task_net, task_sign | Бесконечный |
| `g_display_ctx_mutex` | Структура `g_display_ctx` (4 строки текста, курсор) | task_display | Бесконечный |

**Примечание дизайна:** бесконечный таймаут допустим здесь, потому что:
- Каждая задача держит мьютекс кратко (< 1 мс)
- Ни одна задача не выполняет блокирующие I2C операции, держа мьютекс
- Нет вложенного захвата мьютекса (нет риска дедлока)

### Глобальные флаги и буферы

```c
volatile uint8_t  g_security_alert = 0;     // Установлено task_sign при ошибке валидации
uint8_t          g_last_sig[64] = {0};      // Последняя подпись ECDSA (r || s)
volatile uint8_t  g_last_sig_ready = 0;    // Флаг: подпись готова для ответа USB/HTTP
```

- **`g_security_alert`:** атомарный флаг, читаемый task_io для зажигания LED3
- **`g_last_sig`:** буфер общей подписи; защищён неявным упорядочением FSM (задача подписания пишет, сетевая задача читает после `g_last_sig_ready` = 1)
- **`g_last_sig_ready`:** volatile флаг для обозначения готовности подписи без необходимости семафора

## Критические заметки дизайна

### 1. Почему ранняя инициализация LwIP?

Буферы DMA LwIP должны находиться в регионе памяти со специфичными политиками кеша:
- Write-through для TX буферов (избежать проблем когерентности)
- Cacheable для RX буферов (производительность)

MPU (Memory Protection Unit) должен быть настроен **до** `HAL_Init()` так, чтобы когда `HAL_Init()` включает D-Cache, поведение кеша совпадало с регионами MPU. Это воспроизводится из примеров `lwip_zero` и `lwip-uaid-SSD1306` в STM32CubeH7.

### 2. Глобальные очереди vs. callback

Почему не использовать уведомления задач или callback?

- **Очереди буферизируют:** если сетевая задача ставит в очередь 2 транзакции до того, как задача подписания запустится, обе в очереди (нет потерь).
- **Отделение:** сетевая задача не знает и не волнует о времени задачи подписания; просто ставит в очередь и продолжает.
- **Детерминированное упорядочение:** семантика FIFO ясна и предсказуема.

Для одноразовых сигналов (нажатие кнопки) группы событий эффективнее очередей.

### 3. Почему Task_Security не создана

`task_security.c` реализует альтернативный FSM подписания с mock криптографией. Он связан, но не инстанцирован (`Task_Security_Create()` никогда не вызывается), потому что:
- **Производственная сборка** использует `task_sign.c` с настоящей ECDSA
- **Только аудит/тест** могут инстанцировать `task_security.c` для сравнения поведения подписания
- Ведение обеих связанных позволяет гибкость времени компиляции без загромождения main.c

### 4. Стратегия обработки ошибок

Три фатальных хука ловят неисправляемые ошибки:
- `vApplicationMallocFailedHook()` — куча исчерпана
- `vApplicationStackOverflowHook()` — переполнение стека задачи
- `Error_Handler()` — универсальный ловушка (также вызывается если создание очереди неудачно)

Все три логируют в UART и зацикливаются бесконечно. Это намеренно: во встроенном кошельке обнаружение отказа (логирование) и безопасная остановка важнее, чем изящное восстановление.

### 5. Логирование дисплея при загрузке

`Task_Display_Log()` вызывается на протяжении всего `main.c` **перед тем как** создана задача дисплея. Это работает, потому что:
- `Task_Display_Log()` записывает прямо на SSD1306 и UART
- Это не зависит от работающей задачи; это просто вызов функции
- Ранний диагностический вывод неоценим для отладки зависаний при загрузке
